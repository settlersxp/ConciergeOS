# Prompt Database Table Exposure

## Overview

The `{DATABASE_TABLES}` placeholder resolver can now be configured to expose **only specific database tables** to LLM prompts. This is useful when you want to hide internal or administrative tables (e.g., `test_results`, `PromptVersions`, `PromptGroup`) from the LLM's context while keeping business-critical tables (e.g., `Guests`, `Rooms`, `Reservations`) visible.

## How It Works

Each SQLAlchemy model class can declare a class-level attribute `__expose_in_prompt__`. When this attribute is set to `True`, the corresponding database table will be included in the schema output generated by the `{DATABASE_TABLES}` placeholder.

### Default Exposed Tables

The following models are configured to expose their tables by default:

| Model | Table Name | Description |
|-------|------------|-------------|
| `Room` | `Rooms` | Hotel room definitions |
| `Guest` | `Guests` | Guest directory |
| `Reservation` | `Reservations` | Booking records |

### Hidden Tables (by default)

The following models do **not** have `__expose_in_prompt__` set and are therefore **hidden** from `{DATABASE_TABLES}`:

| Model | Table Name | Description |
|-------|------------|-------------|
| `PerformanceTestResult` | `test_results` | Performance test records |
| `PromptVersion` | `PromptVersions` | Prompt versioning data |
| `PromptGroup` | `PromptGroup` | Prompt group definitions |
| `PromptGroupItem` | `PromptGroupItem` | Prompt group items |
| `PromptGroupSchedule` | `PromptGroupSchedule` | Prompt group schedules |
| `PromptGroupResult` | `PromptGroupResult` | Prompt group execution results |

## Usage

### Exposing a New Table

Add `__expose_in_prompt__ = True` to your model class:

```python
class MyNewModel(Base):
    """Maps to the MyNewTable table."""

    __tablename__ = "MyNewTable"
    __expose_in_prompt__ = True  # <-- Add this line

    id: Mapped[int] = mapped_column(Integer, primary_key=True)
    # ... rest of your fields
```

### Hiding an Existing Table

Simply remove or set `__expose_in_prompt__` to `False`:

```python
class SensitiveModel(Base):
    __tablename__ = "SensitiveTable"
    __expose_in_prompt__ = False  # Explicitly hidden

    id: Mapped[int] = mapped_column(Integer, primary_key=True)
    # ... rest of your fields
```

## Backward Compatibility

If **no models** declare `__expose_in_prompt__`, the resolver falls back to exposing **all** database tables. This ensures existing deployments continue to work without modification.

## How the Resolver Works

The `_resolve_database_tables()` function in `backend/app/services/placeholders.py`:

1. Introspects all database tables via SQLAlchemy's `MetaData.reflect()`.
2. Scans all known model classes for `__expose_in_prompt__ = True`.
3. Filters the schema to include only exposed tables.
4. Generates a structured schema description for inclusion in LLM prompts.

## Logging

When the resolver runs, it logs which tables are being exposed:

```
INFO: Filtering DATABASE_TABLES to exposed models: ['Guests', 'Reservations', 'Rooms']
```

If no models declare the attribute (fallback mode):

```
INFO: No models with __expose_in_prompt__ found — exposing all tables
```

## Adding New Models

When adding new SQLAlchemy models, consider whether the table should be visible to LLM prompts:

- **Business data** (Guests, Rooms, Reservations, Orders, etc.) → Add `__expose_in_prompt__ = True`
- **System/internal data** (Audit logs, test results, internal caches) → Do **not** add the attribute
